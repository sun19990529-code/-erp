# ERP-MES 后续优化建议

> 方案一（WAL 模式）已于 v1.3.1 实施完成。以下为后续可选优化项，按性价比排序。

## 方案二：高频静态数据内存缓存 ⭐⭐⭐

**适用场景：** 产品列表、仓库、工序、角色、权限等"几乎不变"的数据被反复查询。

**实现方式：** 在 `backend/utils/cache.js` 中添加一个简单的 TTL 缓存工具：

```javascript
// backend/utils/cache.js
const store = new Map();

const cache = {
  get: (key) => {
    const item = store.get(key);
    if (!item || Date.now() > item.expireAt) { store.delete(key); return null; }
    return item.data;
  },
  set: (key, data, ttlMs = 30000) => {
    store.set(key, { data, expireAt: Date.now() + ttlMs });
  },
  del: (key) => store.delete(key),        // 数据变更时手动失效
  clear: () => store.clear(),
};

module.exports = cache;
```

**使用示例（在 routes/basic.js 中）：**

```javascript
const cache = require('../utils/cache');

// 读取时走缓存
router.get('/products', (req, res) => {
  const key = `products_${req.query.category || 'all'}`;
  let data = cache.get(key);
  if (!data) {
    data = req.db.all('SELECT * FROM products WHERE status=1 ORDER BY code');
    cache.set(key, data, 30000); // 缓存 30 秒
  }
  res.json({ success: true, data });
});

// 写入时清除相关缓存
router.post('/products', (req, res) => {
  // ... 写入逻辑
  cache.del('products_all'); // 失效缓存
  res.json({ success: true });
});
```

**预期效果：** 减少 60~80% 的数据库查询次数，接口响应提速 3~10 倍。

## 方案三：关键表添加数据库索引 ⭐⭐⭐

**问题：** 订单、库存、生产工单表在查询时若无索引，每次都全表扫描。

**实现：** 在 [database.js](file:///d:/%E9%A1%B9%E7%9B%AE/erp-mes-system/backend/database.js) 的 [createTablesIfNotExist](file:///d:/%E9%A1%B9%E7%9B%AE/erp-mes-system/backend/database.js#36-109) 中补充索引：

```javascript
// 在 createTablesIfNotExist 末尾追加
const indexes = [
  `CREATE INDEX IF NOT EXISTS idx_orders_status ON sales_orders(status)`,
  `CREATE INDEX IF NOT EXISTS idx_orders_created ON sales_orders(created_at DESC)`,
  `CREATE INDEX IF NOT EXISTS idx_inventory_warehouse ON inventory(warehouse_id)`,
  `CREATE INDEX IF NOT EXISTS idx_inventory_product ON inventory(product_id)`,
  `CREATE INDEX IF NOT EXISTS idx_production_status ON production_orders(status)`,
  `CREATE INDEX IF NOT EXISTS idx_production_order ON production_orders(order_id)`,
  `CREATE INDEX IF NOT EXISTS idx_warehouse_orders_status ON inbound_orders(status)`,
  `CREATE INDEX IF NOT EXISTS idx_logs_created ON operation_logs(created_at DESC)`,
];
indexes.forEach(sql => db.exec(sql));
```

**预期效果：** 列表查询从 O(n) 降至 O(log n)，数据量大时效果尤为明显。

## 方案四：API 分页统一化 ⭐⭐

**问题：** 部分接口一次性返回全部数据（如 `/api/products`），数据量大时前端渲染卡顿。

**建议：** 检查并为以下接口补充 `?page=1&pageSize=50` 分页参数：

| 接口 | 当前状态 | 建议 |
|------|---------|------|
| `/api/products` | 全量返回 | 添加分页 |
| `/api/suppliers` | 全量返回 | 添加分页 |
| `/api/customers` | 全量返回 | 添加分页 |
| `/api/production` | 全量返回 | 添加分页+状态过滤 |

## 方案五：前端虚拟滚动（超大表格） ⭐

**适用场景：** 产品列表、库存列表超过 500 条时，DOM 渲染卡顿。

**实现：** 使用 `react-window` 库替换普通列表渲染：

```bash
npm install react-window
```

```jsx
import { FixedSizeList } from 'react-window';

<FixedSizeList height={600} itemCount={data.length} itemSize={52} width="100%">
  {({ index, style }) => <div style={style}>{data[index].name}</div>}
</FixedSizeList>
```

**适用组件：** `ProductManager`、`InventoryView`

## 方案六：定时任务 + 数据归档 ⭐

**问题：** 操作日志、已完成订单随时间累积，查询越来越慢。

**建议：** 添加每月自动归档：

```javascript
// backend/scheduler.js
const archiveOldLogs = db.transaction(() => {
  const cutoff = new Date();
  cutoff.setMonth(cutoff.getMonth() - 3); // 3 个月前
  db.prepare(`DELETE FROM operation_logs WHERE created_at < ?`)
    .run(cutoff.toISOString());
  db.prepare(`PRAGMA wal_checkpoint(TRUNCATE)`).run(); // 压缩 WAL 文件
});

// 每天凌晨 2 点执行
setInterval(() => {
  const now = new Date();
  if (now.getHours() === 2 && now.getMinutes() === 0) archiveOldLogs();
}, 60000);
```
